Paperclip + Hermes + Codex — Hostinger 콘솔로 한 페이지에서 끝내기

단일 컨테이너로 묶인 3-도구 스택을 호스팅어 콘솔의 'URL에서 컴포즈 가져오기'로 1분 만에 배포하고, Codex OAuth 1회 인증으로 영구 사용합니다.

STEP 01

hPanel → VPS → Docker Manager

Hostinger hPanel 왼쪽 사이드바에서 VPS 항목을 열고 Docker Manager 탭으로 이동합니다.

01-1. 이 스택이 하는 일

Paperclip + Hermes + Codex CLI가 단일 컨테이너 안에서 동작합니다. SSH 없이 hPanel 콘솔만으로 배포·운영·업데이트가 완결됩니다.

  1. 01

    Paperclip (port 3100) — 에이전트 오케스트레이터 + Web UI. 태스크를 받아 Hermes 워커에 전달합니다.

  2. 02

    Hermes Dashboard (port 9119) — 세션·스킬·API 키 관리 SPA. 브라우저에서 직접 접근합니다.

  3. 03

    ttyd Terminal (port 4860) — 브라우저 안 Hermes 쉘. 별도 SSH 없이 CLI를 사용할 수 있습니다.

  4. 04

    Codex CLI — Hermes가 태스크별로 spawn하는 LLM 백엔드. OpenAI API를 직접 호출합니다.

paperclip-hermes-codex

01-2. 콘솔 접근 경로

hPanel 로그인 → 왼쪽 사이드바 VPS 클릭 → 서버 선택 → 상단 탭 중 Docker Manager를 클릭합니다. Docker Manager가 보이지 않으면 VPS 플랜이 Docker 지원 여부를 확인하세요.

STEP 02

URL에서 컴포즈 가져오기

Docker Manager의 "URL에서 컴포즈 가져오기" 입력란에 아래 URL을 붙여넣고 가져오기를 클릭합니다.

02-1. Compose URL 
https://raw.githubusercontent.com/dandacompany/paperclip-hermes-codex-on-hostinger/main/docker-compose.console.yml
02-2. console.yml이 하는 일

docker-compose.console.yml은 단독 실행 가능한 self-contained 파일입니다. 메인 컨테이너(paperclip-hermes-codex)와 Tailscale 사이드카를 함께 정의하며, 부팅 시 GitHub Raw에서 tailscale/serve.json을 자동으로 내려받아 3100·9119·4860 포트를 HTTPS로 라우팅합니다. SSH나 별도 파일 업로드 없이 이 URL 한 줄로 전체 스택이 올라옵니다.

STEP 03

환경변수 입력

가져온 컴포즈 아래 환경변수 폼에 아래 값을 입력합니다. OPENAI_API_KEY는 지금 비워두어도 됩니다.

변수필요 시점설명
ADMIN_USERNAME필수Paperclip + Hermes TUI 로그인 ID (예: admin)
ADMIN_EMAIL필수Paperclip 관리자 이메일
ADMIN_PASSWORD필수Paperclip + Hermes TUI 비밀번호 (8자 이상 권장)
ADMIN_NAME선택표시 이름, 기본값 Owner
TS_AUTHKEY필수Tailscale Auth Key — 아래 발급 방법 참고
TS_HOSTNAME선택Tailscale 노드 이름, 기본값 paperclip
PAPERCLIP_PUBLIC_URL나중에 입력STEP 05에서 Tailscale FQDN 확인 후 입력
OPENAI_API_KEY선택Codex OAuth 대신 API 키로 인증할 때 입력. 지금은 비워두기
03-2. Tailscale Auth Key 발급 
# 브라우저에서 아래 URL 열기
https://login.tailscale.com/admin/settings/keys

# "Generate auth key" 클릭 후:
#   - Reusable: 체크
#   - Expiry: 90일 이상
#   - Tags: 선택 사항
# 발급된 tskey-auth-... 값을 TS_AUTHKEY 필드에 붙여넣습니다.
03-3. OPENAI_API_KEY를 지금 비워두는 이유

STEP 08에서 Codex OAuth 1회 브라우저 인증을 완료하면 토큰이 codex-auth named volume에 영구 저장됩니다. 이후 컨테이너를 재시작하거나 이미지를 업데이트해도 재인증이 필요 없습니다. API 키 방식을 선호한다면 OPENAI_API_KEY를 지금 입력해도 되며, Codex CLI 0.122+ 이상은 이 값을 자동으로 auth.json으로 변환합니다.

STEP 04

배포 + 컨테이너 2개 상태 확인

"배포" 버튼을 클릭합니다. Docker Manager가 이미지를 pull하고 두 컨테이너를 순서대로 시작합니다.

04-1. 배포 순서

Tailscale 사이드카가 먼저 healthy 상태가 된 뒤 paperclip-hermes-codex 컨테이너가 시작됩니다 (depends_on: tailscale: condition: service_healthy). 전체 배포는 이미지 크기에 따라 1~3분이 소요됩니다.

04-2. 정상 시작 시 컨테이너 목록 (2개) 
paperclip-hermes-codex-on-hostinger-tailscale-1            running
paperclip-hermes-codex-on-hostinger-paperclip-hermes-codex-1  running
04-3. 확인 기준

컨테이너 2개 모두 running 상태입니다. v0.1 사이드카 구조의 paperclip-init / hermes-init은 v1에서 사라졌습니다 — 권한 설정과 bootstrap이 entrypoint.sh 안으로 통합되었기 때문입니다. 하나라도 exited이면 로그 버튼을 눌러 오류 메시지를 확인하세요.

04-4. 로그 확인 방법

Docker Manager 컨테이너 행 오른쪽 Logs 버튼 클릭 → paperclip-hermes-codex 컨테이너 로그에서 다음 줄을 찾습니다:
[entrypoint] Paperclip bootstrap complete
[entrypoint] Hermes bootstrap complete
[entrypoint] Starting supervisor...

STEP 05

Tailscale FQDN 확인 + PAPERCLIP_PUBLIC_URL 재배포

Tailscale이 노드 FQDN을 할당하면 이를 PAPERCLIP_PUBLIC_URL에 등록하고 재배포합니다. 이 단계를 건너뛰면 로그인 링크에서 403이 납니다.

05-1. Tailscale FQDN 확인 방법 
# 방법 A — Tailscale Admin 콘솔
# https://login.tailscale.com/admin/machines
# → 노드 이름(TS_HOSTNAME)을 찾아 Machine Name(FQDN) 복사
# 예: paperclip.tail1234ab.ts.net

# 방법 B — Docker Manager 로그
# tailscale 컨테이너 로그 → "tailscale up complete" 이후 줄에서
# "https://<fqdn>" 형식으로 노출됩니다.
05-2. PAPERCLIP_PUBLIC_URL 값 형식 
# FQDN이 paperclip.tail1234ab.ts.net 이라면:
PAPERCLIP_PUBLIC_URL=https://paperclip.tail1234ab.ts.net:3100
05-3. 왜 재배포가 필요한가

Paperclip은 better-auth를 사용하며, 허용 origin 목록을 부팅 시점의 PAPERCLIP_PUBLIC_URL 값에서 파생합니다. 이 값이 실제 접근 도메인과 다르면 로그인 시 Invalid origin / 403이 발생합니다. Tailscale FQDN은 첫 배포 이전에는 알 수 없으므로, STEP 04 배포 → FQDN 확인 → 환경변수 갱신 → 재배포 순서가 정상 흐름입니다.

05-4. 확인 기준

PAPERCLIP_PUBLIC_URL이 https://<fqdn>:3100 형식으로 저장되었고 컨테이너 2개가 다시 running 상태입니다.

STEP 06

첫 접속 — Paperclip 로그인

브라우저에서 https://<fqdn>:3100 을 열고 ADMIN_EMAIL + ADMIN_PASSWORD로 로그인합니다. 관리자 계정은 entrypoint가 자동 생성했습니다.

06-1. 접속 URL 
# Tailscale 메시에 연결된 디바이스에서 열기
https://<TS_HOSTNAME>.ts.net:3100

# 예시
https://paperclip.tail1234ab.ts.net:3100
06-2. 관리자 계정 자동 생성 방식

v0.1의 bootstrap-ceo invite token 복사 흐름이 v1에서 사라졌습니다. entrypoint.sh가 부팅 시 ADMIN_EMAIL + ADMIN_PASSWORD env를 읽어 Paperclip에 관리자를 자동으로 가입시키고, 이후 disableSignUp: true로 잠급니다. 로그인 페이지에서 가입 버튼이 없으면 정상입니다.

06-3. 확인 기준

Paperclip 워크스페이스 메인 화면이 표시됩니다. 왼쪽 사이드바에 Agents, Tasks, Settings 메뉴가 보입니다.

STEP 07

Hermes Dashboard 확인

새 탭에서 https://<fqdn>:9119 를 열어 Hermes Dashboard에 접속합니다.

07-1. Hermes Dashboard URL 
https://<TS_HOSTNAME>.ts.net:9119

# 예시
https://paperclip.tail1234ab.ts.net:9119
07-2. Dashboard가 보여주는 것

Hermes Dashboard는 세션 목록, 설치된 스킬 패키지, API 키 설정, 실행 로그를 제공하는 관리 SPA입니다. Codex OAuth 완료 전에도 Dashboard 자체는 접근 가능합니다. ttyd 터미널은 https://<fqdn>:4860에서 열 수 있으며, 접속 시 ADMIN_USERNAME + ADMIN_PASSWORD로 인증합니다.

07-3. 확인 기준

Hermes Dashboard 메인 화면이 표시됩니다. 상단에 Sessions 탭이 보이며, 현재 활성 세션이 없는 상태가 정상입니다.

STEP 08

Codex OAuth 부트스트랩

paperclip-hermes-codex 컨테이너 로그에서 Codex OAuth URL을 찾아 브라우저에서 인증합니다. 이 인증은 단 1회만 필요합니다.

08-1. OAuth가 필요한 이유

Hermes는 태스크 실행 시 Codex CLI를 spawn합니다. Codex CLI가 OpenAI API를 호출하려면 OAuth 토큰 또는 API 키가 있어야 합니다. codex-oauth.sh는 부팅 시 ~/.codex/auth.json을 확인하고, 없으면 codex login --device-auth를 백그라운드로 실행해 URL을 로그에 출력합니다. Paperclip과 Hermes는 OAuth 완료를 기다리지 않고 즉시 시작됩니다.

08-2. 컨테이너 로그에서 OAuth URL 찾기 
# Docker Manager → paperclip-hermes-codex → Logs 버튼 클릭
# 아래와 유사한 줄을 찾습니다:

[codex-oauth] No auth.json found — starting device auth
[codex-oauth] Open this URL to authenticate:
[codex-oauth] https://auth.openai.com/device?user_code=XXXX-XXXX
[codex-oauth] Code: XXXX-XXXX (expires in 15 minutes)
08-3. 브라우저 인증 절차 
# 1. 로그에 표시된 URL을 새 탭에서 엽니다
#    예: https://auth.openai.com/device?user_code=XXXX-XXXX

# 2. ChatGPT 계정으로 로그인 후 "Allow" 클릭

# 3. 콘솔 로그에서 완료 확인:
[codex-oauth] Codex OAuth completed — auth.json saved to /home/node/.codex/auth.json
08-4. OPENAI_API_KEY 대안

OAuth 대신 API 키를 사용하려면 환경변수 OPENAI_API_KEY=sk-...를 추가하고 재배포합니다. Codex CLI 0.122 이상은 이 값을 자동으로 auth.json 형식으로 변환합니다. OAuth와 API 키 중 하나만 있으면 충분하며 두 방식은 혼용 가능합니다.

08-5. 확인 기준

컨테이너 로그에 [codex-oauth] Codex OAuth completed가 출력됩니다. 이후 컨테이너 재시작이나 이미지 업데이트 시 재인증은 필요 없습니다 — 토큰이 codex-auth named volume에 영구 저장되었기 때문입니다.

STEP 09

첫 에이전트 생성 + Test now 통과

Paperclip에서 Hermes Agent 어댑터를 사용하는 CEO 에이전트를 생성하고 Test now 버튼으로 어댑터 환경을 검증합니다.

09-1. 에이전트 생성 절차 
# Paperclip UI (https://<fqdn>:3100)
# 1. 왼쪽 사이드바 → Agents → + New Agent 클릭
# 2. Name: CEO (또는 원하는 이름)
# 3. Adapter: "Hermes Agent" 선택
# 4. Hermes URL: http://localhost:9119  (컨테이너 내부 주소)
# 5. Session Token: Hermes Dashboard → Settings에서 확인
# 6. Save 클릭
09-2. Hermes Agent 어댑터가 동작하는 이유

v1 단일 컨테이너는 hermes 바이너리를 /usr/local/bin에 포함합니다. Paperclip의 어댑터 환경 체크 (Adapter env check)는 같은 컨테이너 PATH에서 바이너리를 즉시 발견하므로 추가 설치 없이 바로 통과됩니다.

09-3. Test now 버튼 클릭 후 기대 결과 
# 에이전트 상세 페이지 → "Test now" 버튼 클릭
# 성공 시 표시:
Adapter env check: passed
09-4. 확인 기준

Test now 버튼 클릭 후 Adapter env check: passed 메시지가 표시됩니다. 오류가 나면 paperclip-hermes-codex 컨테이너 로그에서 [supervisor] hermes dashboard 줄을 확인해 Hermes Dashboard가 실행 중인지 검증합니다.

STEP 10

첫 Task 실행 — 체인 검증

간단한 Task를 생성해 Paperclip → Hermes → Codex → OpenAI 전체 체인이 연결되었는지 확인합니다.

10-1. Task 생성 
# Paperclip UI → Tasks → + New Task
# Title: Hello Chain
# Description: Respond with hello
# Agent: CEO (방금 생성한 에이전트)
# Run 클릭
10-2. 체인 흐름

Paperclip이 태스크를 Hermes Agent에 전달합니다 → Hermes는 codex run으로 Codex CLI를 spawn합니다 → Codex CLI는 auth.json 토큰으로 OpenAI API를 호출합니다 → 결과가 역순으로 전달되어 Paperclip Task 상세 페이지에 반환됩니다.

10-3. 확인 기준

Task 상세 페이지에 에이전트 응답이 표시됩니다. Status가 completed이면 전체 체인 검증이 완료된 것입니다. Status가 failed이면 STEP 08 Codex OAuth가 완료되었는지 다시 확인합니다.

STEP 11

업데이트 흐름

Hostinger Docker Manager 콘솔의 Update 버튼 한 번으로 paperclip, hermes, codex 세 도구의 최신 버전이 반영됩니다.

11-1. GHA nightly 빌드

GitHub Actions가 매일 KST 12:17에 upstream 이미지를 pull해 ghcr.io/dandacompany/paperclip-hermes-codex:latest를 재빌드합니다. Update 버튼은 이 최신 digest를 pull해 컨테이너를 재생성합니다.

11-2. 콘솔 업데이트 절차 
# Docker Manager → 서비스 선택 → Update 버튼 클릭
# 내부 동작:
#   docker compose pull          (최신 이미지 다운로드)
#   docker compose up -d         (컨테이너 재생성)
# 소요 시간: 이미지 크기에 따라 1~2분
11-3. 데이터 보존 원칙

named volume 4개 (paperclip-data, hermes-data, codex-auth, tailscale-state)는 docker compose up -d로 컨테이너가 재생성되어도 그대로 유지됩니다. Codex OAuth 토큰, Hermes 세션·스킬, Paperclip 워크스페이스 데이터, Tailscale 노드 ID가 모두 보존됩니다. 완전히 초기화하려면 Docker Manager → 볼륨 탭에서 4개 볼륨을 수동 삭제합니다.

11-4. 확인 기준

Update 완료 후 컨테이너 2개가 다시 running 상태입니다. Paperclip UI에서 기존 Agents와 Tasks가 그대로 보이면 데이터 보존이 정상입니다. Codex OAuth를 다시 하지 않아도 Task 실행이 정상 동작하면 업데이트가 완료된 것입니다.

11-5. 특정 버전으로 롤백

GHA 빌드는 :latest 외에 :nightly-<N>와 :<sha> 태그도 publish합니다. 문제가 생기면 Docker Manager 환경변수에서 이미지 태그를 ghcr.io/dandacompany/paperclip-hermes-codex:nightly-42 형식으로 지정하고 재배포합니다.
