Hostinger Console · Hermes × Paperclip

Hostinger 콘솔로 Hermes + Paperclip 배포하기

Hostinger VPS 의 Docker Manager 페이지에서 컴포즈 한 벌을 붙여넣고 환경변수만 입력해 Hermes 와 Paperclip 을 사이드카로 띄웁니다. Tailscale 메시 안에서만 인증된 디바이스가 접근하고, 운영도 같은 콘솔에서 끝납니다.

STEP 01

무엇을 설치하는가

Hostinger VPS 의 Docker Manager 콘솔에서 직접 컴포즈 한 벌을 붙여넣어 Hermes 와 Paperclip 을 사이드카로 띄웁니다. SSH 명령은 없습니다.

01-1. 구성

Hostinger 콘솔 UI 만으로 두 시스템과 Tailscale 게이트웨이를 한 프로젝트로 배포한다.

  1. 01

    Hermes Dashboard (9119) · Hermes TUI (4860) · Paperclip (3100) 세 서비스를 한 프로젝트에서 운영한다.

  2. 02

    Tailscale 사이드카가 메시 게이트웨이가 되어, 외부 인터넷에는 한 포트도 노출하지 않는다.

  3. 03

    환경변수와 비밀번호는 Hostinger 콘솔의 변수 입력 폼으로 직접 등록한다.

  4. 04

    재시작·로그·업데이트는 Docker Manager 페이지의 버튼으로 처리한다.

01-2. 이 경로가 맞는 경우

Hostinger VPS 를 처음 다루고 SSH 보다 웹 콘솔이 편한 운영자에게 맞습니다. 이미 SSH 가 익숙하다면 저장소의 install.sh 한 줄 설치를 쓰는 게 더 빠릅니다 (docs/tutorial-hermes-paperclip-on-hostinger/ 참고).

STEP 02

Tailscale auth key 발급

사이드카가 메시에 가입할 때 사용하는 인증 키를 미리 만들어 둡니다.

02-1. 발급 페이지

login.tailscale.com/admin/settings/keys 로 이동해 Generate auth key 를 누릅니다.

02-2. 키 보관

tskey-auth-... 로 시작하는 문자열을 한 번만 화면에 보여줍니다. 4단계에서 환경변수로 입력해야 하므로 안전한 곳에 복사해 둡니다.

STEP 03

Hostinger Docker Manager 진입

hPanel 에서 대상 VPS 의 Docker Manager 페이지로 들어갑니다.

03-1. 경로

hpanel.hostinger.com 에 로그인한 뒤 왼쪽 메뉴 VPS → 사용할 서버 카드의 Manage → 왼쪽 사이드바 Docker Manager 로 이동합니다.

03-2. Traefik 사전 준비 불필요

Traefik 설치는 필요 없습니다. 빈 VPS 에서 바로 시작해도 됩니다. 기존에 다른 Docker Manager 앱이 깔려 있어도 충돌하지 않습니다 (다른 프로젝트 이름·다른 볼륨).

03-3. 빈 프로젝트 화면

처음 진입하면 첫 배포를 시작하세요 안내가 보입니다. 가운데의 검은색 컴포즈 버튼을 누르면 3가지 방식이 펼쳐집니다.

STEP 04

URL 에서 컴포즈 가져오기

컴포즈 드롭다운의 3가지 옵션 중 <strong>URL 에서 컴포즈 가져오기</strong> 를 선택합니다. 이 저장소의 Raw URL 을 붙여넣기만 하면 됩니다.

04-1. 3가지 옵션 중 가운데

수동으로 컴포즈 구성 은 textarea 에 직접 YAML 을 붙여넣는 방식, 원클릭 배포 는 Hostinger 가 제공하는 단일 앱 템플릿입니다. 이 경우 URL 에서 컴포즈 가져오기 가 가장 빠릅니다 — 텍스트 없이 한 줄 입력으로 끝납니다.

04-2. 컴포즈 URL 
https://raw.githubusercontent.com/dandacompany/hermes-paperclip-on-hostinger/main/docker-compose.console.yml
04-3. URL · 프로젝트 이름 입력

URL 필드에 위 Raw URL 을 붙여넣고, 프로젝트 이름 은 hermes-paperclip-on-hostinger 또는 본인이 원하는 짧은 이름으로 입력합니다. 오른쪽 패널에 GitHub · GitLab · Docker Compose 아이콘이 보이는 게 정상 — Hostinger 가 입력한 URL 의 출처를 자동 인식한 표시입니다. 배포 버튼을 누르면 다음 단계(환경변수 입력)로 넘어갑니다.

04-4. 무엇이 들어 있나

단일 self-contained 파일로 init 컨테이너 2개, Hermes TUI · Dashboard, Paperclip, Tailscale 사이드카까지 모두 정의되어 있습니다. Tailscale 사이드카는 부팅 직후 같은 저장소의 tailscale/serve.json 라우팅 파일을 GitHub Raw 에서 받아 적용하므로 추가 파일 업로드가 필요 없습니다. 두 Hostinger 이미지에 platform: linux/amd64 가 박혀 있어 VPS 에서 그대로 부팅됩니다.

STEP 05

환경변수 입력

컴포즈를 가져온 다음 화면에서 6개 변수를 등록합니다. 비밀번호는 미리 32자 랜덤 문자열로 만들어 둡니다.

05-1. 비밀번호를 미리 만든다 (호스트 어디서나 한 번) 
# macOS / Linux 어디서나 한 줄로 32자 랜덤 비밀번호 생성
openssl rand -base64 32 | tr -d '/+=' | head -c 32 ; echo
변수명값 예시용도
ADMIN_USERNAMEhermesHermes TUI Basic Auth 사용자명
ADMIN_NAMEOwnerPaperclip 첫 admin 표시 이름
ADMIN_EMAILyou@example.comPaperclip 첫 로그인 이메일
ADMIN_PASSWORD<32-char random>Hermes TUI · Paperclip 공통 비밀번호
TS_AUTHKEYtskey-auth-...2단계에서 발급한 키
TS_HOSTNAMEhermes-paperclipTailscale 노드 표시 이름
05-2. PAPERCLIP_PUBLIC_URL 은 비워둔다

5단계에서는 이 변수를 등록하지 않거나 임시로 http://localhost:3100 으로 둡니다. Tailscale 메시에 가입한 뒤 진짜 FQDN 을 알게 되면 7단계에서 갱신합니다.

05-3. 변수 입력 후 저장

콘솔의 Save 또는 Add variable 행동으로 6개 모두 등록합니다. 이 시점엔 컨테이너가 아직 부팅 안 됩니다.

STEP 06

Deploy + 컨테이너 상태 확인

Deploy 후 Docker Manager 에 컨테이너 카드 6장이 뜹니다. 4장은 Running, 2장은 Exited 가 정상 상태입니다.

06-1. Deploy

콘솔의 Deploy 또는 Start 버튼을 누릅니다. Hostinger 가 이미지를 pull 하고 컨테이너 6개를 띄웁니다 (약 1~3분, 첫 pull 일수록 더 걸림).

06-2. 정상 부팅 시 상태 — 4 Running + 2 Exited

Running 4 개 · hermes-tui (ttyd 콘솔), hermes-dashboard (Hermes 그래픽 UI), paperclip (Paperclip 워크플로 UI), tailscale (메시 게이트웨이). Exited 2 개 · hermes-init 과 paperclip-init 은 데이터 볼륨 소유권을 한 번 정리하고 종료하도록 설계된 init 컨테이너입니다 — Exited 가 정상 상태이며 종료 코드는 0 입니다. init 이 정상 종료해야 의존성 조건(service_completed_successfully)을 통해 나머지 4 개가 부팅됩니다.

06-3. 첫 부팅에서 Paperclip 이 재시작 루프인 경우

5-2 에서 PAPERCLIP_PUBLIC_URL 을 비웠을 때 정상 동작이지만, 잘못된 값(예: placeholder URL) 을 넣으면 better-auth 가 거부해 재시작 루프에 빠집니다. 그럴 땐 http://localhost:3100 으로 임시 수정 후 Restart.

06-4. Tailscale 사이드카 로그 확인

tailscale 컨테이너 카드의 Logs 를 열어 Switching ipn state Starting -> Running 과 serve: creating a new proxy handler for http://hermes-dashboard:9119 등 3개의 proxy handler 로그를 확인합니다.

STEP 07

메시 FQDN 확인 + 세 인터페이스 접속

Tailscale admin 콘솔에서 노드 도메인을 확정하고, 같은 tailnet 멤버 디바이스에서 Hermes · Paperclip · TUI 세 인터페이스를 차례로 엽니다.

07-1. 메시 노드 확인 — 머신 목록

login.tailscale.com/admin/machines 머신 목록에서 hermes-paperclip (또는 지정한 TS_HOSTNAME) 행을 찾습니다. Address 컬럼의 100.x.x.x 가 mesh IP, LAST SEEN 이 Connected 면 가입 성공입니다.

07-2. 노드 클릭 → 상세 페이지에서 Full domain 확인

목록의 hermes-paperclip 행을 클릭해 상세 페이지로 들어가면 Machine Details 섹션에 Full domain 항목이 보입니다. 이 값(예: hermes-paperclip.tail7b1307.ts.net) 이 메시 멤버 디바이스에서 접속할 때 쓰는 호스트명입니다. Copy 버튼으로 클립보드에 복사해 둡니다.

07-2b. PAPERCLIP_PUBLIC_URL 을 Full domain 으로 즉시 갱신

Paperclip 의 better-auth 는 PAPERCLIP_PUBLIC_URL 한 값에서 trusted origin 6 종을 derive 합니다. 이 값이 http://localhost:3100 인 상태로 메시 도메인에서 접근하면 sign-up 페이지가 Invalid origin / 403 으로 거부합니다. Docker Manager → 환경 탭에서 PAPERCLIP_PUBLIC_URL 을 https://<Full-domain>:3100 (예: https://hermes-paperclip.tail7b1307.ts.net:3100) 으로 저장하고 paperclip 컨테이너만 Restart. 이후 invite URL 접근이 통과됩니다.

07-3. 디바이스 측 Tailscale 준비

접속하려는 노트북·폰에 Tailscale 클라이언트가 설치되어 있고 같은 tailnet 에 가입되어 있어야 합니다. 다른 tailnet 이면 같은 IP·도메인이라도 안 보입니다. 팀원 초대는 admin → Users → Invite users 에서 이메일 발송.

인터페이스메시 URL첫 진입 인증
Hermes Dashboardhttps://<Full-domain>:9119별도 로그인 없음 (Hermes 자체 세션 토큰)
Hermes TUIhttps://<Full-domain>:4860Basic Auth: hermes / ADMIN_PASSWORD
Papercliphttps://<Full-domain>:3100첫 접속은 invite URL 로 admin 등록 (07-5 참고), 이후 sign-in
07-4. Hermes Dashboard 첫 진입

:9119 로 들어가면 Hermes 자체 세션 토큰이 자동 발급되어 별도 로그인 없이 메인 화면이 열립니다. 왼쪽 사이드바의 Sessions · Analytics · Models · Logs · Cron · Skills · Plugins · Profiles · Config · Keys · Documentation 메뉴 중 Keys 에서 LLM provider API 키를 등록하고, Sessions 탭에서 첫 모델 호출을 검증합니다. 하단의 Gateway Status · Active Sessions 표시로 메시징 게이트웨이 상태를 한눈에 확인합니다.

07-5. Paperclip 첫 접속 — Instance setup required 화면

:3100 첫 진입 시 sign-in 폼 대신 Instance setup required 안내가 보입니다. No instance admin exists yet. A bootstrap invite is already active. 와 함께 Paperclip 이 그대로 다음 명령을 추천합니다: pnpm paperclipai auth bootstrap-ceo. 이 화면은 sign-up 폼이 아닌 안내 — 표시된 명령을 실행해 invite URL 을 받아야 가입 흐름이 시작됩니다.

07-6a. paperclip 컨테이너 터미널 열기

Docker Manager 의 프로젝트 상세 페이지에서 컨테이너 카드 목록을 보면 각 카드에 터미널 링크가 있습니다. hermes-paperclip-on-hostinger-paperclip-1 카드의 터미널 링크를 클릭하면 새 탭에 콘솔 창이 열립니다.

07-6b. 콘솔 터미널에서 invite URL 발급

열린 콘솔 창의 프롬프트(node@...:~$)에서 다음 한 줄을 입력합니다 (안내 화면의 pnpm 접두사는 빼고):

paperclipai auth bootstrap-ceo

Hostinger 가 빌드한 이미지는 paperclipai 를 /usr/local/bin/ 에 글로벌 설치해 두어 pnpm 래퍼가 필요 없습니다. pnpm 을 그대로 두면 bash: pnpm: command not found 가 떨어집니다. 성공 시 출력의 Invite URL: https://<Full-domain>:3100/invite/pcp_bootstrap_… 한 줄을 그대로 복사. 07-2b 에서 PAPERCLIP_PUBLIC_URL 을 메시 도메인으로 갱신해 두었기 때문에 invite URL 이 처음부터 메시 도메인으로 발급됩니다.

07-7. invite URL 열고 admin 가입

복사한 invite URL 을 같은 디바이스 브라우저 새 탭에 붙여넣어 엽니다. 좌측 Set up Paperclip 패널이 company · invited by · requested access · invite expires 를 보여주고, 우측 Create your account 폼에 Name · Email · Password 입력 → Create account and continue. 가입 완료 시 invite 가 즉시 소진되고 같은 :3100 URL 은 다음부터 일반 sign-in 폼으로 전환됩니다.

07-8. PAPERCLIP_PUBLIC_URL 의 역할 정리

07-2b 에서 갱신한 PAPERCLIP_PUBLIC_URL 한 값이 다음 네 군데를 동시에 결정합니다. (1) better-auth trusted origin 목록 (메시 도메인 접근 허용), (2) 이메일 magic link / 초대장 URL, (3) OAuth callback URL, (4) Paperclip UI 가 절대 URL 을 생성할 때 prefix. 단일 source of truth — 도메인 변경 시 이 한 줄만 바꾸면 됩니다.

07-9. invite 만료 · 사용됨 시 — 같은 rotate 흐름

Invite not available — This invite may be expired, revoked, or already used. 안내가 뜨면 07-6 의 같은 콘솔 터미널에서 paperclipai auth bootstrap-ceo 를 한 번 더 실행해 새 invite URL 을 받습니다. Logs 뷰에는 CLI 로 rotate 한 invite 가 안 찍히므로 (첫 부팅 시 stdout 만 기록), rotate 는 항상 터미널 흐름으로.

STEP 08

Hermes · Paperclip 첫 사용 + 운영

두 시스템 모두 자체 로그인을 가지고 있습니다. 첫 셋업과 자주 쓰는 운영 동작을 정리합니다.

08-1. Hermes 첫 셋업

Dashboard URL (:9119) 로 들어가 Setup 또는 API Keys 탭에서 LLM provider (Anthropic / OpenAI / OpenRouter / ...) 키와 기본 모델을 등록합니다. TUI 에서 셸로 진행하려면 Docker Manager 의 hermes-tui 컨테이너 카드의 Console 또는 Exec 버튼으로 hermes setup 을 실행합니다.

08-2. Paperclip 첫 로그인

Paperclip URL (:3100) 로 들어가 sign-in 폼에 ADMIN_EMAIL 과 ADMIN_PASSWORD (5단계에서 입력한 값) 를 넣습니다. 첫 로그인 시 admin 계정이 자동 생성됩니다. Paperclip 컨테이너 로그 상단에 pcp_bootstrap_… 로 시작하는 일회용 admin invite URL 이 한 번 찍히는데, 첫 로그인이 끝나면 즉시 소진됩니다 — 외부에 공유하지 마세요.

작업콘솔 동작
재시작Docker Manager → 프로젝트 → Restart
개별 컨테이너 재시작컨테이너 카드의 Restart 버튼
로그 보기컨테이너 카드의 Logs 버튼
환경변수 변경프로젝트의 Environment 탭에서 수정 → Save → Restart
이미지 업데이트프로젝트의 Update 또는 Pull 버튼 (Pull 후 Restart)
비밀번호 회전ADMIN_PASSWORD 새 값 입력 → Save → hermes-tui · paperclip 재시작

08-3. 메시 노드 정리

Hostinger 에서 프로젝트를 지운 뒤 Tailscale admin → Machines 에서 hermes-paperclip 노드를 Remove 해 메시 목록에서 빼냅니다. auth key 는 reusable 이라 그대로 유효합니다.

08-4. 문제가 생기면

Paperclip sign-up 페이지에서 Invalid origin / 403 이 보이면 PAPERCLIP_PUBLIC_URL 이 접근 도메인과 다른 상태입니다 (07-2b 참고). Paperclip 이 부팅 직후 죽으면 PAPERCLIP_PUBLIC_URL 값이 잘못된 URL 형식이라는 뜻입니다 (better-auth 가 invalid base URL 거부). 메시 Full domain 으로 갱신하거나 http://localhost:3100 으로 임시 복구. 그 외 항목은 저장소의 docs/EXPOSURE-tailscale.md 트러블슈팅 절을 따릅니다.
